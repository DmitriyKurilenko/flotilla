# Architecture

> Design document. Written before code. Locks the high-level decisions so
> implementation is mechanical. Lives alongside the codebase and is updated
> when (and only when) a decision changes — never to describe what already
> exists in code.

## 1. Mission

flotilla is a thin orchestrator on top of Docker Compose for **shared-VPS
multi-project deployments**. On a fresh server its installer (`install.sh`)
sets up everything that's needed (Docker, the `proxy` network, Traefik
with Let's Encrypt) in a single interactive run; after that, every
deploy is `flotilla deploy <project>`.

It adds three things to a plain `docker compose up`:

1. **Contract lint** — every project is checked against a small closed
   set of conventions before deploy. The conventions exist because each
   one was paid for in a real production incident.
2. **Unified deploy ritual** — the same `flotilla deploy` works for every
   project regardless of language/framework. The ritual is auditable
   (one binary, one config) instead of a per-project shell script. For
   simple projects (one service, one domain), Traefik labels are
   generated automatically from `project.yml` — the operator never
   writes them by hand.
3. **Introspection** — `flotilla status` answers «is the deployed thing
   actually serving traffic» in one command, by combining Docker state,
   Traefik router registration, and external HTTPS probing.

## 2. Non-goals

flotilla is deliberately **not**:

- A PaaS. There is no web UI, no buildpack magic, no managed databases.
  The operator writes a `compose.yml` and a `project.yml`. flotilla
  deploys them.
- A replacement for Traefik. flotilla does **not** reimplement TLS
  termination, certificate issuance, or reverse-proxy routing — Traefik
  already does these well. `install.sh` deploys Traefik on a fresh VPS
  automatically (see §4.1). After install, certificates appear when
  downstream projects declare Traefik labels — either by the operator's
  hand in `compose.yml`, or generated by flotilla from `https_entrypoint`
  in `project.yml` (§3 + §5 step 0). If an existing Traefik is detected
  on the `proxy` network, `install.sh` reuses it. If a different ingress
  is occupying ports 80/443, `install.sh` fails fast with a pointed
  message and lets the operator decide; there are no flags to suppress
  this check.
- A multi-cluster scheduler. Target topology is **1-5 servers, 1-20
  projects**, single operator or small team. Beyond that scale, k8s is
  the correct answer.
- A build system. Image builds happen inside `docker compose build`,
  which is invoked by flotilla but owned by the compose file.
- A CI/CD runner. flotilla is invoked _by_ CI (typically GitHub Actions
  SSH'ing to the VPS). It is one of the things CI does, not the thing
  that runs CI.
- A secret manager. Secrets live in `.env` files on the VPS, outside
  git. flotilla validates that required ones are set; it never reads
  their values, never re-encrypts them, never rotates them.

These non-goals are how flotilla stays small. If a feature request
crosses one of them, the answer is «not in flotilla — use $other_tool».

## 3. The contract: `project.yml`

A flotilla project is any directory containing a `project.yml` file with
this schema:

```yaml
version: 1                                                   # required
name: crm_prvms                                              # required
domain: crm.prvms.ru                                         # required
path: /opt/crm_prvms                                         # required
description: PRVMS CRM platform (Django + Vue SPA)           # optional
https_entrypoint: web                                        # optional
```

Six fields, four required. `https_entrypoint` accepts a union — either
a bare compose-service name (shortcut) or an object with explicit port:

```yaml
# Shortcut form: port is inferred from compose (errors if ambiguous).
https_entrypoint: web

# Explicit form: required when the service has multiple expose/ports
# entries and flotilla can't pick one.
https_entrypoint:
  service: web
  port: 8000
```

### 3.1 Why each field exists, and why nothing else does

| Field | Why it's here | Why a similar field is _not_ here |
|--|--|--|
| `version` | Schema evolution. When the contract breaks compat, this bumps. The Go parser dispatches on it. | No `apiVersion`/`kind` k8s-style: there is only one kind, and there is no «group» to namespace it under. |
| `name` | Identity in the flotilla catalog. Used in logs, status output, lockfile paths, and as the Traefik router/service base-name when auto-cert is active (`crm_prvms` → `crm-prvms`). | No `description`-as-name fallback; compose's `name:` field is optional and may be absent. Cheaper to require it here than to depend on someone else's optional. |
| `domain` | Primary HTTPS hostname. Needed for `${domain}` template expansion in the built-in smoke test, for matching Traefik routers, for human-readable status, AND for the `Host()` rule when auto-cert is active. | No `domains: []` plural: 95% of projects have one. v2 schema can add multi-domain when there's pain. |
| `path` | Where the project lives on the VPS. flotilla maintains symlinks here (`compose.yml`, `.env.example` → repo source). Server-side concept; compose can't express it. | No `path` default like `/opt/${name}`: explicit > implicit. Magic defaults break when someone needs `/srv/` or `/data/` and can't figure out why their project «isn't found». |
| `description` | Optional. Surfaces in `flotilla status` listing for human readers. | — |
| `https_entrypoint` | Optional. When set, flotilla **generates Traefik labels** for the named service at deploy time (see §3.2 + §5 step 0). Removes the «my compose is perfect but I forgot the Traefik labels and now there's no cert» class of bug. | No multi-router support here: complex routing (path prefixes, multiple services on one host) still uses hand-written labels in compose. The field handles the simple case (one service → one HTTPS endpoint); complex stays manual. |

### 3.2 What is derived from `compose.yml`, not declared

| Derived | How |
|--|--|
| **Required env vars** | Every `${VAR}` reference in `compose.yml` without a `:-default` is required. flotilla parses compose with `docker compose config --no-interpolate` and validates against `.env`. |
| **Optional env vars** | `${VAR:-default}` form. flotilla does not validate. |
| **Service list** | `services:` block of compose. flotilla never re-declares them. |
| **Healthchecks** | `healthcheck:` in compose (per-service). flotilla does not replicate the concept. |
| **Container labels (Traefik)** | When `https_entrypoint` is **not** set in `project.yml` → labels come from `labels:` in compose, hand-written by the operator. flotilla parses `Host(\`…\`)` rules to know which router corresponds to this project. When `https_entrypoint` **is** set → flotilla **generates** the labels into `.flotilla/compose.override.yml` at deploy time; the compose file in git stays untouched. The two modes are mutually exclusive (lint rule L008). |
| **Image build context** | `build:` in compose. flotilla calls `docker compose build`, the compose file owns the «how». |
| **Volumes / networks** | `volumes:` / `networks:` in compose. |
| **Restart policy** | `restart:` in compose. |

This is the central design choice: **compose is the source of truth for
the runtime; project.yml is the source of truth for everything compose
can't express**. The two files do not overlap, by construction.

## 4. On-disk layout of a flotilla project

```
my-project/                          # any directory; could be repo root
                                     # or a subdir of a monorepo
├── project.yml                      # flotilla config (required)
├── compose.yml                      # docker compose (required)
├── .env                             # production secrets (gitignored)
├── .env.example                     # template, committed
├── .flotilla/                       # state + ephemeral artifacts, gitignored
│   ├── state.json                   # last-deploy info (§6.1)
│   └── compose.override.yml         # auto-generated when https_entrypoint
│                                    #   is set; rewritten on every deploy
└── ...                              # the actual app source, build files, etc.
```

`flotilla` operates on _whichever directory has the `project.yml`_. There
is no global registry, no daemon, no state file outside `.flotilla/`.
The CLI's CWD (or explicit path argument) defines the working set.

For a monorepo of deployable units, each unit is its own directory with
its own `project.yml`:

```
my-monorepo/
├── projects/
│   ├── crm/
│   │   ├── project.yml
│   │   ├── compose.yml
│   │   └── .env
│   └── analytics/
│       ├── project.yml
│       └── compose.yml
└── README.md
```

`flotilla deploy projects/crm` deploys exactly that one project.
`flotilla deploy --all` discovers and deploys every `project.yml`
under `/opt/*/project.yml` (the convention path for v0.1; configurable
search paths in v0.2). See §6.

### 4.1 Installation on a fresh VPS

`install.sh` is the single entry point. It does **everything** required
to get a server ready to deploy flotilla projects — installs Docker,
sets up the `proxy` network, deploys Traefik, configures the ACME
resolver, and installs the flotilla CLI binary. The operator never has
to touch Docker, Traefik, or networking by hand on a fresh server.

```bash
curl -fsSL \
  https://raw.githubusercontent.com/DmitriyKurilenko/flotilla/main/install.sh | sh
```

`install.sh` is **interactive by design and has no flags**. The
installer prompts for the email Let's Encrypt should use for expiry
notifications, then proceeds. After it finishes, the operator runs:

```bash
flotilla deploy /opt/my-app
```

…and gets a working HTTPS site. Two commands total.

#### Terminal requirement

`install.sh` requires a tty. Detection: `[ -t 0 ]` (POSIX stdin-is-tty
test). On non-tty (e.g., headless cloud-init without a console, a pipe
that strips terminal allocation), the installer prints a one-line
diagnostic and exits 1 — no silent fallback to a default email that
would later cause Let's Encrypt notifications to disappear into the void.

Because the installer is delivered via `curl | sh`, the script
explicitly reads its interactive input from `/dev/tty` rather than
stdin. This is a known curl-piped-shell pattern.

#### What `install.sh` does, in order

`install.sh` is ~150-200 lines of POSIX shell. It performs the full VPS
bootstrap; the flotilla Go binary is **not** involved until step 8.

1. **Detect OS/arch.** From `uname -s -m` and `/etc/os-release`. Steps 2
   and 7 below have OS-specific behavior; the rest is portable.
2. **Verify / install Docker.** If `docker` binary exists and the daemon
   responds → skip. Else:
   - Debian/Ubuntu → install via the official `https://get.docker.com`
     script (run as `curl -fsSL https://get.docker.com | sh`).
   - Other (RHEL/Alpine/NixOS/macOS/…) → print explicit instruction
     («Install Docker for your platform, then re-run this installer»)
     and exit 1.
3. **Verify Docker Compose plugin.** Must respond to `docker compose
   version`. If missing → print instruction and exit 1.
4. **Prompt for ACME email.** `read -p "Email for Let's Encrypt
   notifications: " ACME_EMAIL` reading from `/dev/tty`. Empty input
   re-prompts. The value is stored in `/opt/traefik/.env` later.
5. **Create the `proxy` Docker network** (idempotent, no-op if exists).
6. **Detect ingress state.**
   - If a container labelled `flotilla.role=ingress` OR a container
     running the `traefik` image is attached to the `proxy` network →
     verify it has `--providers.docker=true` in its command. If not,
     **remove it automatically** (`docker stop && docker rm`) and set
     `INGRESS_STATE=deploy` so a flotilla-compatible Traefik is created.
     If the check passes, log «Traefik already present, reusing» and
     skip to step 8.
   - Else, if any Docker container is bound to host port 80, 443, or 8080
     (`docker ps --format '{{.Ports}}' | grep ':80->' / ':443->' /
     ':8080->'`) → **remove the conflicting containers automatically**
     (`docker stop && docker rm`) and set `INGRESS_STATE=deploy`.
     Port 8080 is included because Traefik's dashboard binds to
     `127.0.0.1:8080`; if occupied, `docker compose up` fails with a
     cryptic Docker networking error. Flotilla assumes it owns the
     ingress on the `proxy` network and cleans up obstacles.
   - Else (ports free, no existing Traefik) → continue.
7. **Deploy Traefik.** The Traefik bundle lives in the flotilla repo at
   `embed/traefik/{compose.yml,project.yml,.env.example}`. `install.sh`
   fetches `compose.yml` via `curl` from
   `https://raw.githubusercontent.com/DmitriyKurilenko/flotilla/main/embed/traefik/...`
   into `/opt/traefik/`. If the download fails (e.g. the repository has
   not yet been published), the script falls back to copying from a
   local `./embed/traefik/compose.yml` when present (dev/testing mode).
   It then writes `ACME_EMAIL` into `/opt/traefik/.env`,
   then `docker compose -f /opt/traefik/compose.yml up -d`. Waits up to
   60s for `localhost:8080/api/http/routers` to respond.
8. **Install the flotilla CLI binary.** Downloads the binary for the
   detected OS/arch from the latest GitHub Release, verifies its
   SHA-256 against `checksums.txt`, places at `/usr/local/bin/flotilla`
   with mode 0755. Verifies with `flotilla --version`.

`install.sh` is **idempotent**. Re-running on a server that already has
flotilla and Traefik just upgrades the binary and re-verifies the
ingress is healthy — it does not destroy or reset anything.

#### When `install.sh` is *not* the right path

- **Air-gapped environment**, no internet from VPS. `install.sh` assumes
  outbound HTTPS to `raw.githubusercontent.com`, `get.docker.com`,
  Docker Hub (Traefik image), and GitHub Releases. v0.1 does not target
  air-gapped installs.

(Operator-with-existing-different-ingress is now handled by the
fail-fast detection in step 6 above. There is no flag to bypass it
because there is no clean «alongside» semantics — flotilla either owns
the ingress on `proxy` or it does not, and that's a structural choice
made by the human, not a runtime flag.)

## 5. The deploy pipeline

`flotilla deploy [path]` runs an explicit, ordered set of steps. Each
step is fail-stop: if a step exits nonzero, the pipeline aborts and the
operator sees the failing step name + stderr. No automatic rollback in
v0.1 (compose has no concept of «previous version» without registry
support; document the manual rollback procedure instead).

| # | Step | What it does | Fails if |
|--|--|--|--|
| 0 | `autocert` | If `project.yml` has `https_entrypoint`, render `.flotilla/compose.override.yml` with the right Traefik labels (router rule `Host(\`${domain}\`)`, websecure entrypoint, `le` cert resolver, loadbalancer port). If absent, skip. | Compose service named in `https_entrypoint` doesn't exist; port can't be inferred and isn't given explicitly. |
| 1 | `parse` | Reads `project.yml`, validates against JSON schema for the declared `version`. | Schema violation. |
| 2 | `lint` | Reads the operator's authored `compose.yml` (NOT the autocert override) and runs the contract checks (§7). The override is flotilla's own generated, known-good output; linting it would make L008 misfire on every auto-cert project. The override matters only to steps 5-7. | Any required check fails. |
| 3 | `env` | Resolves every `${VAR}` in compose; verifies each non-default one is non-empty in `.env`. | Any required var missing or empty. |
| 4 | `symlinks` | Ensures `path/compose.yml`, `path/.env.example` are symlinks to the repo source (or copies them on first run, idempotent). | Permission denied / source missing. |
| 5 | `compose-up` | `docker compose -f compose.yml [-f .flotilla/compose.override.yml] up -d --remove-orphans --force-recreate` in the project dir. | Compose exit nonzero. |
| 6 | `wait-running` | Polls `docker compose ps` until all services are `Up` (or `healthy` if they declare a healthcheck). Timeout default 180s. | Timeout. |
| 7 | `traefik-discover` | Queries Traefik API (via `localhost:8080/api/http/routers`) for routers whose rule contains `Host(\`${domain}\`)`. | None found within 30s, or Traefik API unreachable. Usually one of: Traefik container crashed (`docker logs traefik-traefik-1`), `proxy` network removed, or `install.sh` failed mid-way. Re-running `install.sh` is idempotent and repairs the ingress side. |
| 8 | `smoke` | `GET https://${domain}/`, accepts any status `< 500`. Retries 12× over 2 min. | All retries fail or return 5xx. |

### 5.1 The `--dry` flag

`flotilla deploy --dry [path]` runs steps **0 through 3** and stops.
The pipeline performs only checks; no filesystem mutation, no Docker
call. Specifically:
- Step 0 renders the autocert override **in memory** and feeds it to
  step 2. `.flotilla/compose.override.yml` is **not** written to disk.
- Steps 4-8 are skipped entirely.

Use cases: CI on a PR (validate `project.yml` + `compose.yml` against
the contract before merging); a local «did I break my compose?» check
before pushing.

The pipeline is hardcoded in Go for v0.1. A configurable pipeline (skip
steps, add custom steps) is deferred until at least three real projects
have asked for the same customization.

## 6. The CLI

Two commands in v0.1. Both take a positional path (default: `.`) or
operate across all discovered projects with `--all`.

```
flotilla status [path] [--all]
flotilla deploy [path] [--all] [--keep-going] [--dry]
```

`--all` discovers every `project.yml` at `/opt/*/project.yml` and runs
the command against each, in alphabetical order by `name`. Without
`--keep-going`, the first failure stops the loop; with `--keep-going`,
the command continues and prints a summary at the end.

`--dry` is only valid for `deploy`. See §5.1.

Global flags:
- `--quiet` / `-q` — only errors to stderr, no progress to stdout
- `--json` — machine-readable output for CI/scripts
- `--version` — print flotilla version + commit + project.yml schema versions supported

No `--config` flag. No environment variable overrides for behavior.
flotilla's behavior is fully determined by the `project.yml` +
`compose.yml` + `.env` it finds. This makes it testable and
reproducible.

There is intentionally no `flotilla init` or `flotilla new` command.
Users write `project.yml` and `compose.yml` directly per the schema
(§3) and the conventions (§7). The operator never has to ask flotilla
to scaffold a project — that's what `cp -r` is for, on the rare
occasion it's needed.

### 6.1 `status` output

```
$ flotilla status /opt/crm_prvms
crm_prvms · crm.prvms.ru
  containers     7/7 running (web healthy, db healthy, …)
  traefik        3 routers registered (crm-api · crm-spa · crm-static)
  cert           crm.prvms.ru in acme.json, expires 2026-08-09
  https          GET https://crm.prvms.ru/ → 200 (412ms)
  last-deploy    sha=a7e1026, at 2026-05-11 14:23:01 (yesterday)
```

The `last-deploy` line reads from `path/.flotilla/state.json`, a tiny
lockfile that flotilla writes after every successful deploy.

## 7. The contract: lint rules

Each rule fires a `pass / warn / fail` verdict. `fail` aborts deploy;
`warn` is informational.

### v0.1 rules (closed list — adding to this list requires a CHANGELOG entry)

1. **L001 — Compose syntax** — `docker compose config` exits zero.
2. **L002 — Traefik network attachment** — every service with
   `traefik.enable=true` label has `traefik.docker.network=proxy` AND
   lists `proxy` in its `networks:` section. (Includes labels
   auto-generated from `https_entrypoint`.)
3. **L003 — Router priority** — if more than one router shares the same
   `Host()` rule, each must have an explicit `priority:`. (Prevents
   non-deterministic routing.)
4. **L004 — No `HEALTHCHECK` in Dockerfile** — flotilla scans the
   project's Dockerfile(s) (best-effort regex) and warns on any
   `HEALTHCHECK` line. Healthchecks must live in compose where they can
   be overridden / disabled. (Prevents the trap that hid the
   crm_prvms/rent_django HTTPS issue for a week.)
5. **L005 — Domain matches Traefik labels** — at least one router rule
   in compose (or in the auto-generated override) must contain
   `Host(\`${domain}\`)`. Catches the class of bug where `project.yml`
   and compose drift.
6. **L006 — No localhost healthchecks** — `test:` lines containing
   `localhost`, `0.0.0.0` are warned. Use `127.0.0.1` or the project's
   domain with an explicit Host header. (Prevents busybox-wget IPv6
   grief and Django ALLOWED_HOSTS rejection.)
7. **L007 — Required env not in .env.example** — every `${VAR}` without
   a default that is in `.env` must also be in `.env.example`
   (otherwise no one knows it exists when onboarding a new server).
8. **L008 — `https_entrypoint` vs manual Traefik labels** — if
   `project.yml` has `https_entrypoint: web` AND the compose service
   `web` already carries `traefik.*` labels in `labels:`, fail with a
   pointed message: «pick one ownership model — either remove the
   labels from compose and let flotilla generate them, or remove
   `https_entrypoint` from project.yml and own the labels in compose».
   Mixed ownership is a footgun and we refuse to guess.

This list is **closed** for v0.1. Each new rule needs a documented
incident that motivated it. We are not enforcing best practices we read
about on Hacker News.

## 8. Internals

### 8.1 Language

**Pure Go** for the CLI binary, **bash** for `install.sh`. That's it.

`install.sh` is bash because there is no flotilla binary on a fresh VPS
yet — the script's job is to bootstrap to a state where the Go binary
exists and is on `PATH`. Everything that has tests, branching logic, or
any notion of state lives in Go. The bash script is one isolated file,
~150-200 lines of POSIX shell, with a single purpose: «turn a bare VPS
into a flotilla-ready VPS». Once that's done it's never invoked at
runtime; the CLI carries the load.

External tools (`docker`, `docker compose`, `curl` for `get.docker.com`)
are invoked from the Go CLI via `os/exec`. That is the orchestration
model, not «bash-in-Go». We do not use the Docker Engine SDK
(`github.com/docker/docker`): its dependency graph is enormous and its
ABI compat across Docker versions is fragile; the `docker` CLI is the
stable contract.

Single-binary distribution, cross-compiled by GoReleaser to four
targets (linux/amd64, linux/arm64, darwin/amd64, darwin/arm64). No
runtime dependencies beyond a working Docker installation.

### 8.2 Go module layout

```
flotilla/
├── cmd/
│   └── flotilla/
│       └── main.go                  # CLI entry, command dispatch
├── internal/
│   ├── project/                     # project.yml parsing + validation
│   ├── contract/                    # lint rules (L001…L008)
│   ├── compose/                     # `docker compose` wrapper (os/exec)
│   ├── envcheck/                    # .env / compose interpolation validator
│   ├── traefik/                     # Traefik API client (read-only)
│   ├── autocert/                    # render compose.override.yml from
│   │                                #   https_entrypoint
│   ├── discover/                    # find /opt/*/project.yml for --all
│   ├── deploy/                      # the §5 pipeline as a state machine
│   ├── status/                      # the §6.1 introspection
│   └── log/                         # structured logging + progress
├── schemas/
│   └── project.v1.schema.json       # JSON schema for project.yml v1
├── embed/
│   └── traefik/                     # the Traefik bundle, fetched by
│       ├── compose.yml              #   install.sh via curl
│       ├── project.yml
│       └── .env.example
├── install.sh                       # the bash bootstrap (§4.1)
├── docs/                            # this directory
├── tests/
│   ├── unit/                        # `go test ./...` discovers them
│   └── integration/                 # docker-in-docker, slow, opt-in
└── …
```

There is **no** `internal/install/`, `internal/init/`, `internal/bootstrap/`,
or `lib/bash/` package. VPS bootstrap is bash, period — and it lives at
repo root as `install.sh`, not under `internal/`.

`internal/` everywhere on the Go side because flotilla has no public Go
API in v0.x. Anyone wanting to embed flotilla's logic should call the
CLI.

### 8.3 No daemon, no global state

flotilla is a CLI that runs and exits. There is no `flotilla-agent`, no
systemd unit, no cron timer that flotilla installs. The only on-disk
state is:

- `path/.flotilla/state.json` — last deploy sha, timestamp, summary.
  Written after successful deploys. Read by `flotilla status`.
- `path/.flotilla/compose.override.yml` — auto-generated when
  `https_entrypoint` is set; rewritten on every deploy; safe to delete
  (flotilla regenerates).
- The symlinks flotilla maintains at `path/` per §5 step 4.

No global config in `~/.flotilla/` or `/etc/flotilla/`. Each invocation
is self-contained.

## 9. Distribution

- **Binary releases**: GoReleaser → GitHub Releases, on every git tag.
  Targets: `linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64`.
- **`install.sh`** at the repo root: the full bash installer described
  in §4.1. Not a thin downloader — it does the entire VPS bootstrap.
  Fetched and piped to `sh`:
  ```
  curl -fsSL https://raw.githubusercontent.com/DmitriyKurilenko/flotilla/main/install.sh | sh
  ```
  Idempotent. Re-running upgrades flotilla and re-verifies the ingress;
  never destroys existing project state.
- **Embedded Traefik bundle**: `embed/traefik/{compose.yml, project.yml,
  .env.example}` in the repo, fetched by `install.sh` via curl when
  Traefik isn't already present.
- **Homebrew tap**: deferred until v0.3. Macs are dev machines for
  flotilla, not deployment targets — the CLI on macOS is useful for
  authoring `project.yml` / running `deploy --dry` against local
  project source, but the Docker/Traefik steps of `install.sh` make
  sense only on a server.
- **No Docker image of flotilla itself**: flotilla _operates on_ Docker,
  it doesn't run _inside_ Docker. (You'd need DinD or
  `/var/run/docker.sock` mount — adds complexity for no win.)

## 10. Versioning policy

- `VERSION` file at repo root, single line, single source of truth.
  Read by Makefile / CI / release pipeline. Never duplicated anywhere
  else.
- SemVer with this policy:
  - `0.y.z` — minor bumps can break the CLI. Patch bumps cannot.
  - `1.0.0` — CLI and `project.yml v1` schema are frozen.
  - `2.0.0` — broken compatibility; CLI ships with parsers for v1 and v2
    project.yml simultaneously for one major-version overlap.
- CHANGELOG.md with strict «Added / Changed / Fixed / Deprecated / Removed»
  sections per release. Required PR check that the section has at least
  one bullet under one of those headings.

## 11. Testing strategy

Three tiers:

1. **Unit tests** (Go) — fast, run on every commit. Cover:
   schema parsing, env interpolation, contract rules (one test per rule
   with a positive and a negative fixture), Traefik label parsing,
   autocert rendering. Coverage target: ≥80% on `internal/`.
2. **Integration tests** — Docker-in-Docker in GitHub Actions. Spin up
   a tiny Traefik + Postgres + a hello-world web container, deploy via
   flotilla (both with hand-written labels and with `https_entrypoint`),
   verify HTTPS works end-to-end. Slow (~2-3min), runs on `main` push
   and on PRs touching `internal/deploy`, `internal/compose`, or
   `internal/autocert`.
3. **Dogfooding** — `crm_prvms` and `rent_django` are flotilla's
   continuous canaries. Every release is deployed to them before tag.
   When dogfooding catches a bug, it gets a regression test in tier 1
   or tier 2.

No e2e tests in the «hit a public domain» sense — those belong in the
downstream projects' own CI.

## 12. Roadmap

> Tentative. Each version's scope is _maximum_, not minimum — features
> drop out if they're not pulling weight.

### v0.1 — MVP (target: 2-3 weeks)
- Commands: `status`, `deploy`. Flags: `--all`, `--keep-going`,
  `--dry`, plus the global `--quiet`/`--json`/`--version`.
- 8 contract rules (L001-L008)
- Hardcoded deploy pipeline (steps 0-8 from §5, with `--dry` stopping
  at step 3)
- Single-host (runs ON the VPS, not over SSH from another host)
- `.flotilla/state.json` + ephemeral `.flotilla/compose.override.yml`
- GoReleaser release process
- **`install.sh` (bash) does the full VPS bootstrap**: installs Docker
  on Debian/Ubuntu, creates `proxy` network, deploys Traefik with the
  ACME resolver (email collected interactively), installs the flotilla
  binary. Interactive, no flags, tty-required. Idempotent.
- **Auto-cert via `https_entrypoint`**: simple one-service projects
  declare the field and flotilla generates Traefik labels; no need to
  write them by hand.
- crm_prvms and rent_django migrated to flotilla.

### v0.2 — operator quality-of-life
- `flotilla logs <service>` — tails compose logs of one service
- `flotilla restart-traefik` — convenience wrapper, idempotent. Useful
  when Traefik is in a bad state after manual edits to /opt/traefik.
- `--host <ssh-target>` flag — execute over SSH instead of locally
  (lets one CI workflow deploy to multiple servers without separate steps)
- Configurable `smoke:` list in project.yml (currently the post-deploy
  smoke check is the built-in `GET https://${domain}/` < 500)
- Configurable discovery paths for `--all` (default `/opt/*/project.yml`,
  override in `/etc/flotilla/config.yml`)

### v0.3 — packaging polish
- Homebrew tap for macOS dev workflow

### v1.0 — stability promise
- Frozen CLI, frozen project.yml v1 schema
- Public Go API extracted from `internal/`
- Documented migration path from v0.x

Anything beyond v1.0 is speculative. Don't write it down yet.

## 13. Open questions (still under discussion)

These do _not_ block v0.1 but will affect v0.2+.

- **Multi-host deploy in one command** — when `--host` lands, do we run
  steps sequentially per host, or per-step across hosts (rolling)?
- **Project state in git vs filesystem** — currently `.flotilla/state.json`
  is filesystem-only. Should successful deploys be marked in git via
  annotated tags too? Pros: audit trail; cons: write-back from CI to
  repo is operationally fragile.
- **Backups** — flotilla has no opinion in v0.1. By v1.0 we'll need at
  least a recipe (probably: a sidecar `backup` service in each project's
  compose, with conventions documented in CONVENTIONS.md).
- **Secret rotation** — out of scope for flotilla, but we should
  document recommended patterns (sops? doppler? plain `.env` + secure
  copy?). Recommendation, not implementation.
